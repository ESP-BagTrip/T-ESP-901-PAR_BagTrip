# ADR-0002 — Three pillars unified under Grafana

- **Status**: accepted
- **Date**: 2026-04-27
- **Authors**: Yanis Lounadi
- **Supersedes**: —

## Context

ADR-0001 committed to "the three pillars of observability" without picking
specific products. Each pillar has multiple credible options:

| Pillar | Options | Trade-offs |
|---|---|---|
| Metrics | Prometheus, VictoriaMetrics, Datadog, New Relic | Prom is the de-facto OSS standard, scrape model fits docker-compose, PromQL is a transferable skill. |
| Logs | Loki, Elasticsearch, Splunk, Datadog logs | ELK is heavy (Elasticsearch needs 2+ GB heap), Loki indexes labels not full text — fits container log volumes at our scale. |
| Traces | Tempo, Jaeger, Zipkin, Datadog APM | Jaeger has a strong UI but reintroduces a separate pane; Tempo + Grafana Explore keeps the user in one tool. |
| Alerting | Alertmanager, PagerDuty, OpsGenie | Alertmanager pairs natively with Prometheus and ships routing / inhibition / silences; the SaaS options would have us paying per seat for one user. |

A second axis matters as much as feature parity: **operator surface area**.
Each new product adds a query language, an auth boundary, a config
format, a backup story, and an upgrade path. For a one-engineer M5 deliv,
"one product to learn deeply" beats "five products at 60 % each".

## Decision

We adopt the **Grafana LGTM** stack:

- **Grafana** as the single user-facing pane (queries, dashboards,
  alerts UI, derived-field deeplinks)
- **Loki** for logs (label index → blob store)
- **Tempo** for traces (OTLP receivers → blob store)
- **Mimir** is *not* used; **Prometheus** stays the metrics backend.
  Mimir adds horizontal scaling we don't need yet, and the migration path
  Prom → Mimir is a documented one-liner if traffic ever forces it.

All four products share authoring conventions (YAML config, file-storage
modes for self-hosted, 14–30 d retention defaults), are deployed by the
same Ansible role, and are exposed to ops only via the single
`grafana.bagtrip.fr` hostname.

## Consequences

### Easier
- One auth model (Grafana login → access to everything).
- Cross-pillar correlation is built into the product:
  - Loki → Tempo: every log line carrying a `trace_id` label gains a
    "View trace" button (configured in `grafana_datasources.yml.j2`).
  - Tempo → Loki: every trace exposes a `tracesToLogsV2` deeplink
    that opens the matching log window.
  - Tempo → Prom: span metrics + service-graph generated by Tempo's
    `metrics_generator` and scraped by Prom.
- Dashboards-as-code: `infra/dashboards/*.json` is the source of truth;
  Grafana auto-provisions on container start.
- Same query muscle in PromQL / LogQL / TraceQL — the syntactic
  family is consistent enough that the on-call doesn't context-switch
  language between pillars.

### Harder
- All four backends ship as separate processes; we maintain four sets
  of upgrade notes. (Trade-off vs. Datadog: zero ops vs. €€€/month and
  vendor lock-in.)
- Total RAM budget for the obs stack lands around 2-3 GB. The 26/04
  VPS upgrade gave us 31 GB total — workable.

### Now off-limits
- Mixing UIs (e.g. Kibana for logs while Grafana for metrics). One
  pane, one auth, one set of dashboards.
- Per-pillar storage backends (e.g. ELK for logs only): would
  reintroduce duplicated dashboards / authn / retention work.

## Alternatives considered

| Alternative | Rejected because |
|---|---|
| **ELK / OpenSearch for logs** | Elasticsearch heap pressure on a 31 GB host; full-text indexing buys us little when 95 % of queries are by service / level / trace_id labels. |
| **Jaeger for traces** | Strong UI but separate auth, separate retention story, no built-in tracesToLogs deeplink to Loki. |
| **Datadog / Grafana Cloud (managed)** | M5 rubric explicitly values self-hosting; also hides the configuration we want the jury to see. €€€ per host per month at scale. |
| **VictoriaMetrics instead of Prometheus** | Better long-term storage, but we don't need it yet. The migration is a single drop-in if traffic forces it, so the cost of "switch later" is low. |
| **Mimir instead of Prometheus** | Same logic — adds horizontal scaling we don't need. |
